# Phase 1 Schema Plan

## Purpose

This document plans the Strideo opportunity-centered database schema before any
application tables are created.

Phase 1 remains planning-only:

- No application tables are created.
- No SQL migration is included.
- No application code is written.
- No external racing API is connected.
- No OpenAI integration is connected.

The PRD source of truth is that Strideo measures Opportunities, strategies,
recommended wagers, outcomes, and learning over time. Historical predictions,
odds, recommendations, and verification facts must not be overwritten.

## Schema Principles

- Opportunity is the central aggregate.
- Important generated and observed facts are append-only.
- Current UI state can be derived from immutable events and latest snapshots.
- No table is publicly exposed by default.
- Data API grants must be explicit and added only after RLS policies exist.
- All user-owned data must include `user_id` and RLS.
- Service-role credentials may only be used server-side.
- Browser clients must never call external racing APIs directly.
- Agent outputs must be auditable and linked to job/model/version metadata.

## Opportunity Lifecycle States

Opportunity lifecycle should be event-sourced through `opportunity_events`.
`opportunities.current_state` may store a denormalized latest state for fast UI
filtering, but the history of state transitions must be retained.

Recommended states:

- `candidate`: generated by a strategy/model but not yet eligible for user feed.
- `qualified`: passed minimum score, confidence, odds, and data-quality gates.
- `published`: visible to eligible users through the Opportunity feed.
- `watched`: user saved or followed the Opportunity.
- `sheet_draft`: added to a user's daily bet sheet but not planned.
- `sheet_planned`: user intends to place or monitor the wager.
- `placed`: user marked the wager as placed.
- `closed`: race has started or the Opportunity is no longer actionable.
- `resulted`: official result data has been imported.
- `verified`: performance outcome and ROI have been calculated.
- `voided`: race or wager became invalid, scratched, cancelled, or otherwise not
  gradable.
- `archived`: retained historically but hidden from default active views.

Required `opportunity_events` fields:

- `id`
- `opportunity_id`
- `event_type`
- `from_state`
- `to_state`
- `reason`
- `metadata`
- `created_by_agent_run_id`
- `created_by_user_id`
- `created_at`

Rules:

- State changes insert events.
- No event rows are updated except operational correction fields approved by
  administrators.
- If `current_state` becomes inconsistent, it is rebuilt from events.

## RLS Policy Design

Global access posture:

- No table is granted to `anon` by default.
- No table is exposed through Data API until explicit grants are added.
- All exposed tables must have RLS enabled before grants.
- Service role is server-side only and bypasses RLS only in trusted jobs/routes.

Authenticated user access:

- `profiles`: users can select and update their own profile row.
- `subscriptions`: users can select their own subscription summary; writes are
  server-only.
- `daily_bet_sheets`: users can CRUD rows where `user_id = auth.uid()`.
- `daily_bet_sheet_entries`: users can CRUD rows where `user_id = auth.uid()`.
- `daily_bet_sheet_events`: users can select their own events; inserts are
  server-side or constrained to owned sheets.
- `watchlists`, `alert_preferences`, assistant tables: users can access only
  rows where `user_id = auth.uid()`.

Shared read data:

- Race, entry, odds, strategy, opportunity, and recommendation data may become
  readable to authenticated users only after entitlement rules exist.
- Pro/Elite gating should use trusted `profiles`, `subscriptions`, or
  `app_metadata`, never `user_metadata`.
- Initial MVP can expose read-only racing/opportunity data to authenticated
  users only.

Server-only writes:

- `races`, `race_entries`, `odds_snapshots`, `results`
- `strategies`, `strategy_versions`
- `strategy_matches`, `opportunities`, `opportunity_scores`
- `opportunity_explanations`
- `wager_templates`, `wager_recommendations`
- `recommendation_results`, performance rollups
- `job_runs`, `agent_logs`, `event_log`

Policy implementation notes:

- Add `user_id` indexes for every user-owned table.
- Prefer helper functions in a private schema for entitlement checks if needed.
- Do not place `SECURITY DEFINER` functions in exposed schemas.
- Views used by browser roles should be `security_invoker = true` on Postgres
  versions that support it, or kept out of exposed schemas.

## Index Strategy

Primary feed indexes:

- `races(race_date, track_id, race_number)`
- `races(status, scheduled_post_time)`
- `race_entries(race_id, post_position)`
- `odds_snapshots(race_id, race_entry_id, pool_type, captured_at desc)`
- `opportunities(current_state, generated_at desc)`
- `opportunities(race_id, current_state)`
- `opportunities(strategy_id, current_state, opportunity_score desc)`
- `opportunities(expected_roi desc)` for feed filters
- `strategy_matches(race_id, strategy_id, matched_at desc)`
- `wager_recommendations(opportunity_id, created_at desc)`

User workflow indexes:

- `daily_bet_sheets(user_id, race_date desc)`
- `daily_bet_sheet_entries(user_id, daily_bet_sheet_id)`
- `daily_bet_sheet_entries(user_id, status, created_at desc)`
- `watchlists(user_id, created_at desc)`
- `alert_preferences(user_id)`

Performance indexes:

- `recommendation_results(opportunity_id)`
- `recommendation_results(wager_recommendation_id)`
- `strategy_performance_rollups(strategy_id, strategy_version_id, period_start)`
- `model_performance_rollups(model_version_id, period_start)`

Audit indexes:

- `job_runs(job_type, status, started_at desc)`
- `agent_logs(job_run_id, created_at)`
- `event_log(entity_type, entity_id, created_at desc)`

Notes:

- Use partial indexes for active feed states such as `published`, `qualified`,
  and `closed`.
- Add JSONB GIN indexes only after query patterns are proven; avoid indexing
  every raw feature payload prematurely.

## Partition Strategy

Partition candidates:

- `odds_snapshots`: partition by `captured_at` month or by `race_date` if copied
  onto the row.
- `event_log`: partition by `created_at` month.
- `agent_logs`: partition by `created_at` month.
- `recommendation_results`: partition by `created_at` or `race_date` after
  volume is understood.

Recommended MVP approach:

- Start with monthly range partitions for `odds_snapshots`.
- Add monthly range partitions for `event_log` and `agent_logs` before real
  ingestion/agent volume begins.
- Keep `races`, `race_entries`, `opportunities`, and user tables unpartitioned
  initially, with strong indexes.

Rules:

- Partition keys must be present in write paths.
- Keep foreign key strategy simple in MVP; avoid partitioning every related
  table until volume proves it is necessary.
- Plan maintenance for partition creation before each month.

## Data Retention And Archive Strategy

Hot data:

- Current and upcoming race cards.
- Recent odds snapshots.
- Active and recently settled Opportunities.
- Current user bet sheets.

Warm data:

- Last 12-24 months of Opportunities, recommendations, results, and performance.
- Strategy/model rollups for product analytics.

Archive data:

- Raw odds snapshots older than the active analytics window.
- Large raw provider payloads.
- Agent logs and verbose feature payloads after rollups are computed.

Retention recommendations:

- Never delete core Opportunity, recommendation, result, or performance facts
  unless required for legal/compliance reasons.
- Compress or archive high-volume raw provider payloads after hashing and
  normalization.
- Keep rollups indefinitely for strategy learning.
- Define a future user-data deletion policy before public launch.

## Agent Write/Read Ownership By Table

Race Data Agent:

- Writes: `tracks`, `horses`, `jockeys`, `trainers`, `races`, `race_entries`,
  `odds_snapshots`, `results`, `job_runs`, `agent_logs`.
- Reads: provider mapping tables, prior race/entry identity records.

Horse Intelligence Agent:

- Writes: future horse feature snapshots, `agent_logs`.
- Reads: `horses`, `race_entries`, `results`, `odds_snapshots`.

Prediction Agent:

- Writes: future prediction outputs, `opportunity_scores`, `agent_logs`.
- Reads: race/entry/history feature tables.

Value Agent:

- Writes: `opportunity_scores`, `agent_logs`.
- Reads: predictions, `odds_snapshots`, `race_entries`.

Strategy Agent:

- Writes: `strategy_matches`, `opportunities`, `opportunity_events`,
  `agent_logs`.
- Reads: `strategies`, `strategy_versions`, scores, odds, race context.

Wager Construction Agent:

- Writes: `wager_recommendations`, `opportunity_events`, `agent_logs`.
- Reads: `opportunities`, `strategy_matches`, odds, wager templates.

Race Analyst Agent:

- Writes: `opportunity_explanations`, `agent_logs`.
- Reads: Opportunity context and supporting facts.

Alert Agent:

- Writes: future alerts and `event_log`.
- Reads: `opportunities`, user alert preferences, subscriptions.

Bet Sheet Agent:

- Writes: user-owned bet sheet suggestions only through server-side user context.
- Reads: `opportunities`, `wager_recommendations`, user preferences.

Performance Verification Agent:

- Writes: `recommendation_results`, `opportunity_performance`,
  `strategy_performance_rollups`, `model_performance_rollups`, `agent_logs`.
- Reads: results, Opportunities, recommendations, strategy/model versions.

Strideo Assistant Agent:

- Writes: future assistant session/action tables.
- Reads: only through structured, authorization-aware server tools.

## API Access Boundaries

Browser/Supabase client:

- Uses publishable key only.
- May read authenticated/entitled data through RLS and explicit grants.
- May write only user-owned workflow data through RLS, such as bet sheet rows.
- Never calls The Racing API directly.
- Never calls OpenAI directly.

Server routes/actions/jobs:

- Call external racing APIs.
- Call OpenAI or other AI providers.
- Use service role only where needed for trusted ingestion, verification, or
  administrative writes.
- Validate user session and entitlement before returning user-specific data.

Data API exposure:

- Default exposure remains off.
- Grants to `authenticated` are added table-by-table only after RLS policies are
  reviewed.
- `anon` should remain blocked for MVP private surfaces.

Assistant access:

- No free-form SQL from assistant prompts.
- Assistant tools query approved server-side functions/modules.
- Assistant actions are logged and linked to user/session context.

## Risks And Open Questions

Risks:

- Building UI before schema/RLS review could create insecure or throwaway flows.
- Odds snapshot volume could grow faster than expected without partitioning.
- Strategy/model reruns could confuse users if version history is not clear.
- Assistant tooling could become a data exfiltration path without strict tools.
- Subscription entitlements are not yet defined in enough detail for Pro/Elite.
- Provider identity matching may create duplicate horses, jockeys, or trainers.
- Exotic wager structures can become complex if stored only as untyped JSON.

Open questions:

- What is the first racing provider endpoint and import cadence?
- Should odds snapshots store copied `race_date` for easier partition pruning?
- What is the initial Opportunity Score formula?
- Which tables should be readable by Free users versus Pro users?
- Should wager legs be normalized in a separate table for MVP, or stored as JSON
  until exotic wager complexity increases?
- How should official result corrections be surfaced in performance history?
- What is the minimum viable subscription/entitlement schema before public beta?
- What retention window is acceptable for raw provider payloads?
- Which agent execution surface is preferred first: Vercel cron, Supabase Edge
  Functions, or manual admin jobs?

## Phase 1 Exit Criteria

Phase 1 planning is ready for migration work when:

- This schema plan is reviewed and approved.
- The first migration boundary is selected.
- RLS policies are drafted before SQL implementation.
- Data API grants are listed explicitly.
- Partition and index choices are accepted for MVP volume.
- No application tables are created until the migration PR is requested.
